# shadowmap reference data (Mathematica jleGroup originals)

Reference values for `jlegroup.shadowmap`, generated by running the ORIGINAL
Mathematica implementation — there is no reference publication for this
module; the package defines the functionality:

- `jleGroup`shadowMap`` **4.1.4** (shadowMap_4.1.4.m, dated 2013-11-20)
- `jleGroup`timeAndCoords`` **7.5.6** (its date/GMST/sexagesimal dependency)

run in **Mathematica 14.0** (wolframscript, macOS ARM) from the installed
package tree `/Applications/Wolfram.app/Contents/AddOns/ExtraPackages/jleGroup`
on 2026-07-14 by `generate_references.wls` (committed here):

    wolframscript -file generate_references.wls <outdir>

All numbers are machine doubles printed via CForm.  The script generates more
files than are committed; only the ones the Python tests use are kept (the
Python implementation replaces the original's time/sun machinery with astropy
and the original's continent database with Natural Earth, so the dropped
outputs — projected continents, night polygons, low-precision sun internals —
are no longer oracles; regenerate them with the script if ever needed).

## Committed files and their role

| file | contents (original function) | role in tests |
|---|---|---|
| `tc_dates.csv` | date string -> MJD, GMST hours (tcDateTimeStringToMJD, tcGMST) | exact MJD parsing pin; GMST cross-check at the dUT1 class (the original treats UTC as UT1) |
| `tc_ra.csv`, `tc_dec.csv` | sexagesimal -> radians (tcHMSstringToRadians, tcDecStringToRadians) | exact pins of star_coord's unit conventions |
| `sun_position.csv` | smSunPosn positions (Kepler and equation-of-center) | cross-check of astropy get_sun at the old theory's arcminute class |
| `globe_cases.csv` | four smGlobe configurations G1-G4 with every derived scalar (MJD, GMST, sun RA/Dec, sub-star point, night-cap center) | sub-star / sub-solar cross-checks at the superseded chain's accuracy class |
| `smdist.csv` | smDist(b arcsec, D au) -> Earth radii | exact (au-rounding tolerance, below) |
| `smoffset.csv` | smOffset scenarios (retrograde x addOffToStar, negative-RA-offset branch) | exact for b/PA; au-rounding tolerance on the time shift |
| `directions.csv` | findDirection compass names (trailing spaces intended) | byte-exact |
| `G*_tracks.csv` | drawtracks line endpoints (near edge / center+far multiline / dashed error lines) | exact (the track construction is ported verbatim) |

Globe cases: G1 Pluto-like June event (star 17h45, -22.5 deg, h=0); G2
high-north star, h=18, negative dist; G3 equatorial star at the 1988 equinox,
track miss (subdist > 1); G4 southern star, occulter larger than Earth
(subdist < -1).

## Defects of the original established with this data

1. **Night-shadow placement** (showShadow): the x component of the
   antisolar-direction transform uses `+ sin(sundec) sin(viewlat)` where the
   view rotation requires a minus sign, so the night cap is drawn centered at
   the wrong view longitude whenever both view latitude and solar declination
   are nonzero.  Measured on the original's own drawn boundary (by rotating
   it back to earth coordinates and evaluating the true sun altitude, which
   should equal -horizon_angle everywhere): G1 ±4.7 deg around 0; G2 -23.4 to
   -12.6 (target -18); G3 exact (view lat 0); G4 -37.9 to +25.9 (target -6).
   The Python implementation centers the cap on the true antisolar point and
   is validated by that same invariant instead (exact to the spherical
   model's machine precision).

2. **Obliquity timescale** (smSunPosn): the obliquity argument is computed as
   (MJD difference)/(86400 * 36525) — days divided by a century of seconds —
   freezing the obliquity at its J2000 value.  Both defects are superseded
   wholesale by astropy (get_sun / ITRS) in the Python implementation.

Constants note: the original hardcodes R_earth = 6378.14 km and au =
1.496e8 km.  The port takes both from `jlegroup.physicalData`
(BODIES["Earth"].radius_km carries the same 6378.14; AU_KM is the DE-405
1.49597870691e8, a +1.42e-5 relative difference).  Tests of smDist and of
the smOffset time shift budget for that 1.42e-5 accordingly.

Coastlines: the Python implementation bundles the public-domain Natural
Earth 1:110m coastline (naturalearthdata.com; 134 polylines, 5128 vertices;
fetched 2026-07-14 from the nvkelso/natural-earth-vector GitHub repository)
as `src/jlegroup/data/ne_110m_coastline.npz`, replacing the Mathematica
package's embedded hand-digitized outlines.
